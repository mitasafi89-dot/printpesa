# 02 — Game Engine & Math

## 1. The shared curve
A single authoritative curve (`BTC/KES` styled) is generated by the Game Engine and broadcast to
**all** connected players. There is exactly one curve; everyone sees identical ticks at the same time.

### 1.1 Tick stream
- The engine emits a tick every **100–200 ms** (5–10 ticks/sec).
- Each tick: `{ t: epoch_ms, rate: number, delta: number }`.
- `rate` is the displayed quote (e.g. `0.2138`); the chart plots `rate` oscillating around a baseline.

### 1.2 Smooth, green-biased wave generation
The curve must look **very smooth (wave-like)** with **more green (upward) highs appearing more
often**. We achieve this with a layered model, then render with spline smoothing:

```
value(t) = baseline
         + A1 * sin(2π f1 t + φ1)       // slow swell
         + A2 * sin(2π f2 t + φ2)       // medium wave
         + A3 * noise_smoothed(t)       // gentle randomness (OU process)
         + drift_bias                   // small positive bias → more green highs
```
- **Ornstein–Uhlenbeck (OU)** mean-reverting noise keeps it smooth and bounded (no spikes/gaps).
- A small positive `drift_bias` and asymmetric amplitude (taller upswings) make **green peaks more
  frequent and more prominent** than red troughs — purely visual styling.
- **Client rendering:** Catmull-Rom / cubic-spline interpolation between ticks → glassy-smooth line;
  area fill green above baseline, red below, with glow.

> Visual bias ≠ payout odds. The look of the curve is decoupled from the win probability, which is
> governed by the house-edge calibration in §3. This is standard and must be disclosed in game rules.

## 2. Round / position lifecycle
This game uses a **per-player position on a shared curve** (continuous market model), matching the
screenshot (BUY/SELL + 10s auto-sell + Live P&L + manual SELL).

```
IDLE ──BUY/SELL──▶ OPEN ──(price moves; Live P&L updates each tick)──▶
       │                              │
       │                              ├── player taps SELL ──▶ SETTLED (manual cashout)
       │                              └── auto-sell timer hits 0 ─▶ SETTLED (auto)
```

1. **Open:** player chooses `stake (≥50)`, `direction (BUY=up / SELL=down)`, `duration (default 10s)`.
   - Stake is **debited immediately** (atomic wallet hold).
   - `entry_rate` = current tick rate; `entry_t` recorded; server-seed bound (see §4).
2. **Live:** each tick the engine computes unrealized P&L and a live multiplier (1.0×–5.0×).
3. **Close:** at manual SELL or timer expiry → outcome locked, payout credited atomically.

### 2.1 Multiplier & payout formula
Let `r0 = entry_rate`, `r1 = exit_rate`, `dir = +1 (BUY) | -1 (SELL)`.
```
signed_move = dir * (r1 - r0) / scale          // scale = configured sensitivity
gross_mult  = clamp(1 + k * max(signed_move, 0), 0, MAX_MULT)   // MAX_MULT = 5.0
payout      = (signed_move > 0) ? round(stake * gross_mult, 2) : 0
pnl         = payout - stake
```
- If the curve moved in your predicted direction → you win `stake × multiplier` (capped ×5.0).
- If it moved against you (or not enough) → you lose the stake.
- `k` and `scale` are tuned so the **expected RTP = 25%** (see §3).

## 3. House edge / RTP calibration (75% edge → 25% RTP)
**Target:** over many positions, `Σ payouts / Σ stakes = 0.25` (house keeps 75%).

The engine enforces this with an **outcome controller** layered on the displayed curve:
- For each position, the engine draws the *settlement outcome* from a distribution calibrated by
  Monte-Carlo simulation so that `E[payout/stake] = RTP`.
- Mechanically: the probability that a position closes "in the money", and the multiplier
  distribution when it does, are parameterised `(p_win, mult_dist)` such that
  `p_win * E[mult | win] = RTP = 0.25`.
- Example calibration (illustrative): `p_win = 0.20`, `E[mult|win] = 1.25` → `0.20 * 1.25 = 0.25`. ✅
- The **displayed curve remains continuous and shared**; the controller nudges the realised
  exit-rate (within the smooth envelope) so the aggregate matches RTP without visible discontinuities.

### 3.1 Configurability (Admin)
- `house_edge` (default 0.75), `max_multiplier` (5.0), `min_stake` (50), `tick_rate_ms`,
  `default_duration_s` (10), `drift_bias`, `volatility`. All live-editable; changes audit-logged.
- A built-in **RTP monitor** compares realised RTP vs target over rolling windows and alerts on drift.

## 4. Provable fairness (simple server RNG, MVP)
- Each round/position is bound to a **server seed**. Per game-day the engine generates a
  `server_seed` and publishes `sha256(server_seed)` (the *commitment*) in advance.
- Per position: `outcome_rng = HMAC_SHA256(server_seed, position_id || nonce)` drives the settlement
  draw. The position record stores `server_seed_hash`, `nonce`, and final `rate` path hash.
- At end of day (or on rotation) the `server_seed` is **revealed**, letting anyone verify
  `sha256(revealed) == published_hash` and recompute outcomes. (Client-seed mixing is a later upgrade.)

## 5. Anti-abuse rules
- Server is authoritative; client P&L is display-only and re-validated on settle.
- One open position per player at a time (MVP) to keep accounting simple.
- Settlement is idempotent (idempotency key = `position_id`); double-settle impossible.
- Min stake 50, max stake & max exposure per player configurable; daily loss limits (responsible gaming).
