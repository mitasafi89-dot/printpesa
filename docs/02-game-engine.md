# 02 — Game Engine & Math

> This document reflects the **implemented** engine (`apps/engine`, `packages/shared`).
> All claims here are covered by automated tests (smoothness proof, Monte-Carlo RTP proof,
> lifecycle + idempotency tests).

## 1. The shared curve
A single authoritative curve (styled `BTC/KES`) is generated by the engine and broadcast to **all**
players. There is exactly one curve; everyone sees identical ticks at the same time.

### 1.1 Tick stream
- One tick every `tick_rate_ms` (default 150ms, ~6–7/sec): `{ t, rate, delta }`.
- `value(t)` is a pure function of the daily server seed and time `t`, so all clients render an
  identical curve and every outcome is reproducible.

### 1.2 Smooth, green-biased generation (no pointed curves)
The curve is built by **band-limited Fourier synthesis**: a fixed set of sinusoids whose
frequencies (geometric spread), amplitudes (≈1/f) and phases are derived deterministically from the
seed, summed, then passed through `tanh` for bounding, with a small additive `drift_bias` for green
dominance:
```
raw(t)  = Σ_k amp_k · sin(2π·freq_k·t + phase_k) / ampNorm
value(t)= tanh(raw(t)·volatility + drift_bias)        ∈ (-1, 1)
rate(t) = base + 0.25·value(t)
```
- `sin(...)` and `tanh(...)` are **C-infinity** (infinitely differentiable) ⇒ it is mathematically
  impossible for the curve to contain a pointed peak (a C0 cusp).
- **Proof (test):** for a C2 function the discrete second difference scales ~`f''·dt²`, so refining
  `dt` ×10 shrinks `max|Δ²|` ~×100. The smoothness test asserts this quadratic shrink for the curve
  and validates the test discriminates by confirming a triangle wave only shrinks ~×10.
- Green dominance is a visual property (drift bias); it does **not** bias fairness — see §3.

## 2. Round / position lifecycle
Per-player positions settle against the shared curve (matches the screenshot: BUY/SELL + auto-sell
timer + Live P&L + manual SELL).
```
IDLE ──BUY/SELL──▶ OPEN ──(Live P&L each tick)──▶
  │                          ├── winner taps SELL ─▶ SETTLED (locks current ≤ committed multiplier)
  │                          └── auto-sell timer ──▶ SETTLED (win → committed multiplier; loss → 0)
```
1. **Open:** validate `stake ∈ [min,max]`, atomically **debit stake** (see docs/07), record the
   position. The outcome is committed in memory from the curve but **never persisted before settle**
   (so a player cannot read their result via RLS).
2. **Live:** each tick the engine emits a live multiplier / P&L.
3. **Close:** at manual SELL (winners only) or auto-sell expiry → payout credited atomically.

### 2.1 Settlement rule (curve-determined)
Let `dir = +1 (BUY) | −1 (SELL)`, `entry = value(t_open)`, `exit = value(t_open + duration)`:
```
signedMove = dir · (exit − entry)
win   ⇔ signedMove ≥ τ_dir
mult  = min(1 + gain_dir · signedMove, max_multiplier)   on win
payout= round(stake · mult)  on win, else 0   (loss = −stake)
```

## 3. RATIFIED design decision — exact 75% edge requires loss-gating
**Impossibility:** *one shared curve* + *honest loss-cutting manual cashout* + *exact 75% edge*
cannot coexist. A cashout-invariant P&L must be a martingale, whose mean cannot be both 0 at open and
−0.75·stake at expiry. **Therefore losses are gated to expiry.**

**Ratified model (implemented):**
- Winners' Live P&L rises monotonically (C² quintic smoothstep) to the committed multiplier;
  **manual SELL locks the current value (≤ committed)** — so early exit can only *reduce* payout.
- **Losing positions cannot be cashed out**; the loss is realised at the auto-sell timer.
This keeps the edge exact and non-gameable. (Alternative SELL-anytime makes the edge
behaviour-dependent and was rejected because it violates the fixed 75% requirement.)

### 3.1 RTP calibration (house edge 75% → RTP 25%)
`(τ_dir, gain_dir)` are calibrated **per direction** at engine start so hold-to-expiry RTP equals
`1 − house_edge`:
1. Sample the `signedMove` distribution (per direction) from the seeded curve.
2. `τ_dir` = the `(1 − targetWinRate)` quantile ⇒ `P(win) = targetWinRate` (default 12.5%).
3. Solve `gain_dir` by bisection so `mean(min(1+gain·move, max) | win) = RTP / targetWinRate`,
   accounting for the cap.
Calibrating per direction neutralises the curve's visual green bias, so BUY and SELL have equal RTP.

**Verified (Monte-Carlo, held-out samples):** aggregate **RTP 24.80% → edge 75.20%**
(Buy 25.30% / Sell 24.66%, win-rate ≈12.4%), generalising across multiple daily seeds.
`house_edge`, `max_multiplier`, `targetWinRate`, etc. are admin-configurable; a runtime RTP monitor
compares realised vs target.

## 4. Provable fairness (simple server seed, MVP)
- Per game-day the engine commits `sha256(server_seed)` in advance and reveals `server_seed` later.
- `value(t)` and the per-direction thresholds `τ_dir` derive solely from the committed seed, so given
  the revealed seed + public `(entry_time, direction)` anyone can recompute the move, threshold and
  exact outcome. (Client-seed mixing is a later upgrade.)
- The committed outcome is **not** stored on the open position row, preventing pre-settle leakage.

## 5. Anti-abuse
- Server-authoritative; client P&L is display-only and re-validated on settle.
- Atomic, idempotent settlement keyed by `position_id` (DB RPC) — double-settle impossible
  (verified under concurrent attempts).
- Min/max stake enforced; one open position per player (MVP) keeps accounting simple.
- Losses cannot be cashed out early (edge integrity); responsible-gaming limits in docs/14.
